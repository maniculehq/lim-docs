This page is the cookbook your coding agent should read end-to-end. It maps directly to the bundled `limrun-ios` skill: the same instructions an agent gets when you run `lim skills install`. By the end of this guide your agent will be building an iOS app on Limrun, driving the simulator, and posting a preview link back to a PR.

If you'd rather have the skill installed natively into Claude Code, Cursor, or Codex (so it's discoverable by the agent on its own), see [Install the skill](#install-the-skill-into-your-agent) at the bottom.

You are an iOS build-and-test operator working with Limrun. Read /agents/cli.md end-to-end, then follow this loop for every task that involves running an iOS app:

## Install the CLI

```bash
npm install --global @limrun/cli
```

Confirm with `lim --help`. You should see topics `ios`, `android`, `xcode`, `asset`, `session`, and `skills`. Requires Node.js 20 LTS or newer.

## Authenticate

The CLI resolves credentials in this order: `--api-key` flag, then `LIM_API_KEY` (works with shell exports and `.env`), then `~/.lim/config.yaml`.

```bash
# Option A: browser login
lim login

# Option B: env var (best for cloud agents and containers)
export LIM_API_KEY=lim_...
```

`lim login` opens `https://console.limrun.com/authn/login` in your browser and waits on `http://localhost:32412/authn/callback` for the redirect. The returned key is written to `~/.lim/config.yaml` with mode `0600`.

## Create an iOS + Xcode pair

A single command provisions both the simulator and the attached Xcode sandbox. Use `--reuse-if-exists` plus a stable `--label` set so subsequent runs (e.g. the same PR, the same issue, the same agent session) return the same instance.

```bash
lim ios create --xcode \
  --reuse-if-exists \
  --label issue=<ISSUE_ID> \
  --label repo=<REPO_NAME> \
  --label agent=<AGENT_NAME>
```

The CLI prints the IDs, the **Signed Stream URL** (open it in a browser to watch the simulator live), and the Xcode sandbox URL. Share the stream URL with the user when you start so they can watch.

### Useful create flags

| Flag | Use |
|---|---|
| `--xcode` | Attach an Xcode build sandbox to the simulator (required for `lim xcode build`). |
| `--reuse-if-exists` | Return the existing instance with matching labels and region instead of provisioning a new one. |
| `--label key=value` | Repeatable. Used by `--reuse-if-exists` and for cleanup queries. |
| `--region eu-north1` | Pin the instance to a region. |
| `--inactivity-timeout 10m` | Auto-terminate after N of no activity. `1m`, `10m`, `3h`. Default is 3m. |
| `--hard-timeout 3h` | Force-terminate after a hard ceiling. Default `0` means no hard cap. |
| `--install ./build.app` | Local app file to upload and install at boot. Repeatable. |
| `--install-asset appstore/Expo-Go-54.0.6.tar.gz` | Pre-installed asset from Asset Storage. Repeatable. |
| `--model iphone\|ipad\|watch` | Device model. Defaults to `iphone`. |
| `--rm` | Delete the instance on CLI process exit (useful in one-shot agent invocations). |

## Stay connected with persistent sessions

By default, every `lim ios` subcommand opens a fresh WebSocket to the instance, runs one action, and disconnects. That's fine for occasional commands, but multi-step agent loops benefit from a persistent connection.

```bash
lim session start            # opens a background daemon connected to the last instance
lim session status           # see active sessions
lim session stop             # close the session (or --all)
```

The daemon listens on a per-instance Unix socket under `~/.lim/sessions/<hash>/s.sock`. Subsequent `lim ios` commands detect it and reuse the existing connection, dropping per-call latency from ~500ms (TCP+TLS+WS handshake) to single-digit milliseconds.

The console's Connect modal (cable icon next to a running instance) is the GUI equivalent. It surfaces one-click MCP installers for Replit, Cursor, and Claude Code that match the wiring on [MCP for controlling instances](/agents/mcp).

## Build

Always use `lim xcode build`. Never local `xcodebuild`. The CLI delta-syncs the working directory to the remote Xcode sandbox, runs `xcodebuild` server-side, streams the logs back, and on success auto-installs and auto-launches the app on the attached simulator.

```bash
lim xcode build .
```

For multi-scheme projects or workspaces:

```bash
lim xcode build . --workspace MyApp.xcworkspace --scheme MyApp
```

### Key build flags

| Flag | Use |
|---|---|
| `--scheme MyApp` | Xcode scheme. |
| `--workspace path.xcworkspace` | Use a workspace file. |
| `--project path.xcodeproj` | Use a project file (alternative to workspace). |
| `--sdk iphonesimulator\|iphoneos\|watchsimulator\|watchos` | Target SDK. |
| `--upload <asset_name>` | Upload the build artifact to Asset Storage under this name. Used for preview links. |
| `--signed-upload-url <url>` | Upload the artifact to a pre-signed URL you provide (S3, GCS, etc). |
| `--certificate-p12 path.p12` `--certificate-password ...` `--provisioning-profile path.mobileprovision` | Sign for real-device builds. |
| `--additional-file local=remote` | Sync an extra file (e.g. `~/.netrc`) into the sandbox each build. Repeatable. |
| `--ignore 'regex'` | Exclude paths from the sync. Repeatable. |
| `--basis-cache-dir path` `--max-patch-bytes N` | Tune the xdelta3 client cache. |

The default sync ignores `.git`, `.DS_Store`, the basis cache, and everything matched by your project's `.gitignore` (with `.xcconfig` kept regardless). Build artifacts (`build/`, `.build/`, `DerivedData/`, `xcuserdata/`, `.dSYM/`) are also excluded by default.

## Verify state before acting

Always read state before issuing the next action. There is no `sleep` needed between actions and the read; the calls are synchronous on the device side.

```bash
lim ios element-tree
```

The element tree is the authoritative source of truth for the UI (labels, accessibility IDs, types, frames). For large screens it can be sizable, so pipe through `grep` or `jq`:

```bash
lim ios element-tree --json | jq '.[] | select(.AXLabel | contains("Continue"))'
```

For visual-only checks, fall back to screenshots:

```bash
lim ios screenshot ./out.png
```

## Drive the UI

Prefer accessibility identifiers, then labels, then coordinates as a last resort. The first form is stable across UI redesigns; the last form is brittle.

```bash
lim ios tap-element --ax-unique-id startButton
lim ios tap-element --ax-label "Save"
lim ios tap-element --type Button --ax-label "Done"
lim ios tap 201 450
```

`tap-element` accepts: `--ax-unique-id`, `--ax-label`, `--ax-label-contains`, `--type`, `--title`, `--title-contains`, `--ax-value`. At least one is required; combining them narrows the match.

Text input, keys, scrolling:

```bash
lim ios type "hello world" --enter
lim ios press-key enter --modifier shift
lim ios scroll down --amount 300
lim ios open-url "https://apple.com"
```

### Batch actions for precise timing

When you need a sequence executed without round-trip latency between steps, use `lim ios perform`. The server runs the whole batch in one shot and stops on the first failure.

```bash
lim ios perform \
  --action type=tap,x=100,y=200 \
  --action "type=typeText,text=Hello World"

lim ios perform \
  --action type=wait,durationMs=1000 \
  --action type=pressKey,key=enter
```

Or from a file:

```bash
lim ios perform --file ./actions.yaml
```

```yaml actions.yaml
- type: tapElement
  selector: { AXLabel: "Continue" }
- type: wait
  durationMs: 500
- type: typeText
  text: "hello"
  pressEnter: true
```

Supported action `type`s: `tap`, `tapElement`, `incrementElement`, `decrementElement`, `setElementValue`, `typeText`, `pressKey`, `scroll`, `toggleKeyboard`, `openUrl`, `setOrientation`, `wait`, `touchDown`, `touchMove`, `touchUp`, `keyDown`, `keyUp`, `buttonDown`, `buttonUp`. Hardware buttons: `home`, `lock`, `side`, `applePay`, `softwareKeyboard`.

## Apps and logs

```bash
lim ios list-apps                              # bundle IDs of every installed app
lim ios launch-app com.example.MyApp
lim ios launch-app com.example.MyApp --mode RelaunchIfRunning
lim ios terminate-app com.example.MyApp
lim ios install-app ./MyApp.app                # or a URL
lim ios install-app https://...               # remote download, server-side
```

Streaming logs (Ctrl-C to stop), or tail the last N lines:

```bash
lim ios app-log com.example.MyApp --follow
lim ios app-log com.example.MyApp --tail 200
lim ios syslog
```

## Record a demo video

For anything involving motion (animations, gestures, gameplay), prefer recording over screenshots. Always include a demo video on the PR you ship.

```bash
lim ios record start
# ... drive the UI ...
lim ios record stop -o /tmp/demo.mp4
```

`--quality 5` (default) through `--quality 10` trades file size for fidelity. Pass `--presigned-url` instead of `-o` to upload the recording directly to your own bucket.

## Finalize: upload the build and post a preview link

When the work is done and you're opening a PR, build with `--upload` so the build artifact lands in Asset Storage, then construct a preview URL the reviewer can open from any browser.

```bash
ASSET_NAME="${PR_NUMBER:-$(date +%s)}.zip"
lim xcode build . --upload "${ASSET_NAME}"
```

The preview URL:

```
https://console.limrun.com/preview?asset=${ASSET_NAME}&platform=ios
```

Always include this link in your final message to the user (and in the PR description).

## Clean up

```bash
lim ios delete
```

Without an ID, this removes the last iOS instance the CLI created for you. Its ID is read from `~/.lim/last-instances.json`. To clean up many instances created across an agent session, use a label selector:

```bash
lim ios list --label-selector "agent=cursor,issue=LIM-34"
# then delete each by ID, or terminate them in batch from the console
```

## Gotchas

- **Instance ID is optional.** The CLI remembers the last created instance. Pass `--id` explicitly only when you're juggling multiple.
- **`element-tree` can be large.** Pipe through `grep` / `jq` rather than dumping the full tree into the agent's context window.
- **No sleep needed between actions.** `tap-element` blocks until the gesture completes; the next `element-tree` reads up-to-date state.
- **Build errors are your job.** If `lim xcode build` fails, read the error output, fix the code, and rebuild. Don't push the failure back to the user.
- **Bundle ID discovery.** If you don't know the bundle ID, check the Xcode project files or run `lim ios list-apps` after a successful build.
- **`lim ios delete` defaults to the last instance.** If you've been creating instances in a loop, double-check `lim ios list` before deleting.

## Install the skill into your agent

`lim skills install` copies the bundled skill into the native skills directory for Claude Code, Cursor, or Codex so the agent discovers it without being prompted. The skill file ships inside `@limrun/cli` itself; read the canonical version at [`typescript-sdk/packages/cli/skills/limrun-ios/SKILL.md`](https://github.com/limrun-inc/typescript-sdk/blob/main/packages/cli/skills/limrun-ios/SKILL.md) if you want to see what your agent will be reading.

```bash
lim skills install                         # interactive (recommended first time)
lim skills install --agents claude --scope project
lim skills install --agents cursor --agents codex --skills limrun-ios --scope global
```

| Flag | Values | Use |
|---|---|---|
| `--agents` | `claude`, `cursor`, `codex` (repeatable) | Which agent directory to write into. Interactive if omitted. |
| `--skills` | `limrun-ios`, `limrun-detox` (repeatable) | Which bundled skill to install. Defaults to `limrun-ios`. |
| `--scope` | `project`, `global` | Project scope = current directory's `.claude/` (or equivalent). Global = your home dir. |
| `--force` |   | Overwrite an existing skill file. |
