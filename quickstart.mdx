Limrun gives you remote, programmable access to iOS simulators, Android emulators, and Xcode build sandboxes. This page walks the iOS+Xcode path because it's the most common starting point. By the end of this guide you'll have built an iOS app on Limrun's hosted Xcode and seen it running on a remote simulator in your browser.

```text wrap

2. Authenticate: run `lim login` (browser flow, writes to `~/.lim/config.yaml`) or set `LIM_API_KEY` in `.env` and confirm `.env` is in `.gitignore`.

5. After a successful build, drive the UI with `lim ios element-tree` and `lim ios tap-element --ax-label "<label>"`. Tear down with `lim ios delete`.

Never call local `xcodebuild`. Never spin up a local simulator. Use the `lim` CLI for every build and every device interaction. For deeper context, read /agents/cli.md.
```

## Prerequisites

- **A Limrun API key.** Sign up at [console.limrun.com](https://console.limrun.com), open **Settings → API Keys**, click **Create Token**, copy the value (prefixed `lim_`), and save it to a `.env` file at the root of your project. Make sure `.env` is in `.gitignore`.
  <Frame>
    <img src="/images/console/13-api-keys.png" alt="Settings → API Keys page on console.limrun.com with a Create Token button and a list of existing keys" />
  </Frame>
  ```bash .env
  LIM_API_KEY=lim_...
  ```
- **The `lim` CLI.** Install [`@limrun/cli`](https://www.npmjs.com/package/@limrun/cli) from npm. Requires Node.js 20 LTS or newer.
  ```bash
  npm install --global @limrun/cli
  lim --help   # should list topics: ios, android, xcode, asset, session, skills
  ```

## Set up the build

<Steps>
  <Step title="Authenticate">
    Run `lim login` to open the console in your browser. The CLI listens on `http://localhost:32412` for the callback and writes the returned key to `~/.lim/config.yaml` (mode `0600`).

    ```bash
    lim login
    ```

    For CI, devcontainers, or cloud agents that can't open a browser, export `LIM_API_KEY` instead. The CLI and every SDK pick it up automatically.

    ```bash
    export LIM_API_KEY=lim_...
    ```
  </Step>

  <Step title="Create an iOS instance with Xcode attached">
    A single `lim ios create --xcode` call provisions the simulator and an attached Xcode build sandbox, then prints the IDs and stream URL when the instance reaches `ready`. The `--reuse-if-exists` plus `--label` pattern is the safe shape for re-runs: the second call returns the same instance instead of provisioning a new one.

    ```bash
    lim ios create --xcode --reuse-if-exists --label project=quickstart
    ```

    Open the **Signed Stream URL** from the output in any browser to watch the simulator live. The signed URL embeds its own short-lived token, so no API key is needed in the browser.
  </Step>

  <Step title="Build and run your app">
    From inside an iOS project on your local disk, run:

    ```bash
    lim xcode build .
    ```

    For multi-scheme projects pass `--scheme <name>` and either `--workspace <path>` or `--project <path>`.

    Don't have an iOS project handy? Clone the sample app:

    ```bash
    git clone https://github.com/limrun-inc/sample-native-app.git
    cd sample-native-app
    lim xcode build .
    ```

    Build logs stream back to your terminal as they happen. On success, the build artifact auto-installs and auto-launches on the attached simulator. The browser tab pointed at the Signed Stream URL updates to your app's first screen in under a second.

    <Frame>
      <img src="/images/console/05-ios-stream-booting.png" alt="Live iOS simulator streaming inside the Limrun console with the iOS home screen visible" />
    </Frame>
  </Step>
</Steps>

## Skip the CLI for your first run

If you'd rather poke at an instance before writing any code, [console.limrun.com](https://console.limrun.com) has a **Playground** that creates and streams an instance from a dropdown. Pick an OS, pick an app to preinstall (or none), hit **Start**, and a live simulator drops into the page. Switch to the CLI once you want builds (`lim xcode build`), labels, or scripted lifecycle.

## Drive the app

Once the build lands, you can read state and drive the UI directly from your shell. Most commands default to the last-created instance, so you rarely need to pass `--id`.

```bash
lim ios screenshot ./screen.png
lim ios element-tree
lim ios tap-element --ax-label "Continue"
lim ios type "Hello"
```

The element tree is the source of truth for what the UI shows: labels, accessibility IDs, types, frames. Reach for it before every action; it's faster and more reliable than visual inspection.

## Handle build failures

If `lim xcode build` fails, the same log stream contains the diagnostic. The remote `xcodebuild` is the real Xcode toolchain, so error messages, signing problems, and scheme issues all surface exactly as they would locally. Fix the code, rerun `lim xcode build .`, and the delta-sync layer only ships the bytes that changed.

## Tear down

When you're done, delete the instance. Without an ID, `lim ios delete` removes the last iOS instance the CLI created for you (the pointer is in `~/.lim/last-instances.json`).

```bash
lim ios delete
```
